## Overview

A modern multi-tenant PDF research assistant application. It provides authenticated workspaces (Notebooks) where users can upload PDFs securely. The backend processes the documents asynchronously, generating embeddings and writing them to a vector database for Retrieval-Augmented Generation (RAG) during chat interactions.

## Components

### Client (Next.js Application)
- **Purpose:** Frontend application providing landing page, file upload, auth integration, and AI chat interface.
- **Location:** `client/`
- **Dependencies:** React, Next.js, Clerk, Zustand, Appwrite, Axios.

### Server API
- **Purpose:** Core API for uploads, starting batch processes, managing user metadata/notebooks via Prisma, and facilitating RAG /chat endpoint.
- **Location:** `server/` (Entry: `server/index.js`)
- **Dependencies:** Express, Prisma, Appwrite, Qdrant Client, BullMQ.

### Background Worker
- **Purpose:** Pulls jobs from BullMQ queue to extract text from a PDF, chunk it via LangChain, generate vector embeddings, and upsert them to Qdrant.
- **Location:** `server/worker.js`
- **Dependencies:** BullMQ, LangChain, Qdrant Client, Appwrite.

## Data Flow

1. User uploads a PDF via the Client.
2. The Server API saves the file to Appwrite storage and creates a processing job in BullMQ (via Valkey/Redis).
3. The Worker processes the queued job, fetches the file, chunks the text, creates embeddings, and stores them in Qdrant, tracking metadata via Appwrite/Postgres.
4. User chats with the assistant. The Client sends the message to the Server API.
5. The Server API converts the message to embeddings, performs a vector search in Qdrant, and sends context to an LLM provider to retrieve the answer.
